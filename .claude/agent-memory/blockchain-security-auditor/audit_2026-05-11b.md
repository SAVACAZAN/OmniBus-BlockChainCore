**All prior OPEN issues remain OPEN** — no fixes applied since 2026-05-11 first pass.

**SEV-01 re-confirmed (HIGH)**: rpc_server.zig:7635 `std.mem.eql(u8, got, token)` still used. `constantTimeEql` helper exists at line 7606 in the same file but is NOT called in `isAuthorized`. One-line fix: replace the call.

**SEV-NEW-04 (sync.zig) re-confirmed in detail**: applyBlock at lines 466-514 does only 2 checks (height sequence + timestamp > 0). No PoW hash verification, no merkle check, no canonical prev_hash check. Block hash reconstructed from peer-supplied `merkle_root` bytes. Any peer appends zero-work blocks.

**SEV-NEW-01 (encrypted_p2p.zig) re-confirmed in detail**: HMAC is not symmetric — Alice derives a different value than Bob. Test at line 148 masks this by copying alice's privkey into bob before handshake. In production all sessions fail to decrypt; P2P encryption is non-functional.

**NEW HIGH**:
- NEW-2026-11-01: p2p.zig:2363-2376 `checkRateLimit()` defined and tested (line 1963) but NEVER called in handleInboundPeer receive loop or dispatchMessage; 100 msg/s and 10 MB/s limits are dead code; single peer can flood at unlimited rate
- NEW-2026-11-02: mempool.zig:541-570 `replaceByNonce()` skips `self.verifier` callback; attacker with knowledge of victim (from_address, nonce) can replace victim TX with arbitrary destination without valid sig

**NEW MEDIUM**:
- NEW-2026-11-04: transaction.zig:319-424 calculateHash has no chain_id or genesis-hash in domain; valid testnet TX replays on mainnet if nonce matches (first reported 2026-05-10 as NEW-11, re-confirmed)
- NEW-2026-11-06: p2p.zig:576-603 PeerConnection.recv allocates `allocator.alloc(u8, header.payload_len)` from u32 wire value before size check; 4 GB OOM per malicious peer
- NEW-2026-11-07: mempool.zig:541-570 replaceByNonce also skips TX_MAX_BYTES, TX_MIN_FEE_SAT, and isValid() checks (distinct from sig bypass issue)
