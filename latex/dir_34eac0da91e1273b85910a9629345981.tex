ECDSA {\ttfamily sign()} allows providing {\ttfamily extra\+Entropy}, which switches sig generation to hedged mode.

By default, ECDSA signatures are generated deterministically, following \href{https://www.rfc-editor.org/rfc/rfc6979}{\texttt{RFC 6979}}. However, purely deterministic signatures are vulnerable to fault attacks. Newer signature schemes, such as BIP340 schnorr, switched to hedged signatures because of this. Hedging is basically incorporating some randomness into sig generation process.

For more info, check out \href{https://paulmillr.com/posts/deterministic-signatures/}{\texttt{Deterministic signatures are not your friends}}, \href{https://www.rfc-editor.org/rfc/rfc6979}{\texttt{RFC 6979}} section 3.\+6, and \href{https://datatracker.ietf.org/doc/draft-irtf-cfrg-det-sigs-with-noise/}{\texttt{cfrg-\/det-\/sigs-\/with-\/noise draft}}.\hypertarget{dir_34eac0da91e1273b85910a9629345981_autotoc_md4600}{}\doxysubsubsubsection{\texorpdfstring{Consensus-\/friendliness vs e-\/voting}{Consensus-\/friendliness vs e-\/voting}}\label{dir_34eac0da91e1273b85910a9629345981_autotoc_md4600}

In ed25519, there is an ability to choose between consensus-\/friendliness vs e-\/voting mode.


\begin{DoxyItemize}
\item {\ttfamily zip215:\+ true} (default) uses the more permissive, \href{https://hdevalence.ca/blog/2020-10-04-its-25519am}{\texttt{consensus-\/friendly}} verification rules defined in \href{https://zips.z.cash/zip-0215}{\texttt{ZIP215}}.
\item {\ttfamily zip215:\+ false} enforces strict \href{https://www.rfc-editor.org/rfc/rfc8032}{\texttt{RFC 8032}} /\+ \href{https://csrc.nist.gov/publications/detail/fips/186/5/final}{\texttt{FIPS 186-\/5}} verification and adds SBS-\/based non-\/repudiation, which is useful for:\+
\begin{DoxyItemize}
\item {\bfseries{Contract signing:\+}} prevents a signer from later claiming they signed a different document
\item {\bfseries{E-\/voting:\+}} stops voters from choosing keys that let them repudiate their vote
\item {\bfseries{Blockchains:\+}} avoids signatures valid for multiple transactions (e.\+g., amount X also validating amount Y)
\end{DoxyItemize}
\end{DoxyItemize}

Both modes have SUF-\/\+CMA (strong unforgeability under chosen message attacks). \doxylink{namespace_we}{We}\textquotesingle{}re targetting algorithmic constant time. {\itshape JIT-\/compiler} and {\itshape Garbage Collector} make "{}constant time"{} extremely hard to achieve \href{https://en.wikipedia.org/wiki/Timing_attack}{\texttt{timing attack}} resistance in a scripting language. Use low-\/level languages instead of JS /\+ WASM if your goal is absolute security.

The library mostly uses Uint8\+Arrays and bigints.


\begin{DoxyItemize}
\item Uint8\+Arrays have {\ttfamily .fill(0)} which instructs to fill content with zeroes but there are no guarantees in JS
\item bigints are immutable and don\textquotesingle{}t have a method to zeroize their content:\+ a user needs to wait until the next garbage collection cycle
\item hex strings are also immutable:\+ there is no way to zeroize them
\item {\ttfamily await fn()} will write all internal variables to memory. With async functions there are no guarantees when the code chunk would be executed. Which means attacker can have plenty of time to read data from memory.
\end{DoxyItemize}

This means some secrets could stay in memory longer than anticipated. However, if an attacker can read application memory, it\textquotesingle{}s doomed anyway:\+ there is no way to guarantee anything about zeroizing sensitive data without complex tests-\/suite which will dump process memory and verify that there is no sensitive data left. For JS it means testing all browsers (including mobile). Supported node.\+js versions:\+


\begin{DoxyItemize}
\item v2 (2025-\/08):\+ v20.\+19+ (ESM-\/only)
\item v1 (2023-\/04):\+ v14.\+21+ (ESM \& CJS)
\end{DoxyItemize}\hypertarget{dir_34eac0da91e1273b85910a9629345981_autotoc_md4627}{}\doxysubsubsection{\texorpdfstring{Changelog of curves v1 to curves v2}{Changelog of curves v1 to curves v2}}\label{dir_34eac0da91e1273b85910a9629345981_autotoc_md4627}
v2 massively simplifies internals, improves security, reduces bundle size and lays path for the future. \doxylink{namespace_we}{We} tried to keep v2 as much backwards-\/compatible as possible.

To simplify upgrading, upgrade first to curves 1.\+9.\+x. It would show deprecations in vscode-\/like text editor. Fix them first.


\begin{DoxyItemize}
\item The package is now ESM-\/only. ESM can finally be loaded from common.\+js on node v20.\+19+
\item {\ttfamily .js} extension must be used for all modules
\begin{DoxyItemize}
\item Old:\+ {\ttfamily @noble/\+curves/\+ed25519}
\item New:\+ {\ttfamily @noble/\+curves/\+ed25519.\+js}
\item This simplifies working in browsers natively without transpilers
\end{DoxyItemize}
\end{DoxyItemize}

New features:\+


\begin{DoxyItemize}
\item webcrypto:\+ create friendly noble-\/like wrapper over built-\/in Web\+Crypto
\item oprf:\+ implement RFC 9497 OPRFs (oblivious pseudorandom functions)
\begin{DoxyItemize}
\item \doxylink{namespace_we}{We} support p256, p384, p521, ristretto255 and decaf448
\end{DoxyItemize}
\item weierstrass, edwards:\+ add {\ttfamily is\+Valid\+Secret\+Key}, {\ttfamily is\+Valid\+Public\+Key}
\item misc:\+ add Brainpool curves:\+ brainpool\+P256r1, brainpool\+P384r1, brainpool\+P512r1
\end{DoxyItemize}

Changes:\+


\begin{DoxyItemize}
\item Most methods now expect Uint8\+Array, string hex inputs are prohibited
\begin{DoxyItemize}
\item The change simplifies reasoning, improves security and reduces malleability
\item {\ttfamily Point.\+from\+Hex} now expects string-\/only hex inputs, use {\ttfamily Point.\+from\+Bytes} for Uint8\+Array
\end{DoxyItemize}
\item Many methods were renamed, upgrade to curves v1.\+9 first to highlight deprecated old names
\item Breaking changes of ECDSA (secp256k1, p256, p384...):\+
\begin{DoxyItemize}
\item To bring back old behavior, pass {\ttfamily \{ prehash:\+ false, lowS:\+ false \}} to sign /\+ verify
\item sign, verify:\+ Switch to {\bfseries{prehashed messages}}. Instead of message\+Hash, the methods now expect unhashed message. To bring back old behavior, use option {\ttfamily \{prehash:\+ false\}}
\item sign, verify:\+ Switch to {\bfseries{lowS signatures}} by default. This change doesn\textquotesingle{}t affect secp256k1, which has been using lowS since beginning. To bring back old behavior, use option {\ttfamily \{lowS:\+ false\}}
\item sign, verify:\+ Switch to {\bfseries{Uint8\+Array signatures}} (format:\+ \textquotesingle{}compact\textquotesingle{}) by default.
\item verify:\+ {\bfseries{der format must be explicitly specified}} in {\ttfamily \{format:\+ \textquotesingle{}der\textquotesingle{}\}}. This reduces malleability
\item verify:\+ {\bfseries{prohibit Signature-\/instance}} signature. User must now always do {\ttfamily signature.\+to\+Bytes()}
\end{DoxyItemize}
\item Breaking changes of BLS signatures (bls12-\/381, bn254):\+
\begin{DoxyItemize}
\item Move get\+Public\+Key, sign, verify, sign\+Short\+Signature etc into two new namespaces:\+ bls.\+long\+Signatures (G1 pubkeys, G2 sigs) and bls.\+short\+Signatures (G1 sigs, G2 pubkeys).
\item verify\+Batch now expects array of inputs {\ttfamily \{message:\+ ..., public\+Key:\+ ...\}\+[]\+}
\end{DoxyItemize}
\item Curve changes:\+
\begin{DoxyItemize}
\item Massively simplify curve creation, split it into point creation \& sig generator creation
\item New methods are {\ttfamily weierstrass() + ecdsa()} /\+ {\ttfamily edwards() + eddsa()}
\item weierstrass /\+ edwards expect simplified curve params (Fp became p)
\item ecdsa /\+ eddsa expect Point class and hash
\item Remove unnecessary Fn argument in {\ttfamily pippenger}
\end{DoxyItemize}
\item modular changes:\+
\begin{DoxyItemize}
\item Field\+\#from\+Bytes() now validates elements to be in 0..order-\/1 range
\end{DoxyItemize}
\item Massively improve error messages, make them more descriptive
\end{DoxyItemize}

Previously, the library was split into single-\/feature packages \href{https://github.com/paulmillr/noble-secp256k1}{\texttt{noble-\/secp256k1}}, \href{https://github.com/paulmillr/noble-ed25519}{\texttt{noble-\/ed25519}} and \href{https://github.com/paulmillr/noble-bls12-381}{\texttt{noble-\/bls12-\/381}}.

Curves continue their original work. The single-\/feature packages changed their direction towards providing minimal 5kb implementations of cryptography, which means they have less features. Separate bls package had been deprecated.

secp256k1:\+


\begin{DoxyItemize}
\item {\ttfamily get\+Public\+Key}:\+ defaults to compressed sigs (use second argument to adjust)
\item {\ttfamily sign}:\+ renamed options {\ttfamily canonical} =\texorpdfstring{$>$}{>} {\ttfamily lowS}; {\ttfamily der} =\texorpdfstring{$>$}{>} {\ttfamily format:\+ \textquotesingle{}der\textquotesingle{}}
\item {\ttfamily verify}:\+ renamed option {\ttfamily strict} =\texorpdfstring{$>$}{>} {\ttfamily lowS}
\item {\ttfamily get\+Shared\+Secret}:\+ defaults to compressed sigs (use third argument to adjust)
\item {\ttfamily recover\+Public\+Key(msg, sig, rec)} was changed to {\ttfamily sig.\+recover\+Public\+Key(msg)}
\item {\ttfamily Point} (2d xy) has been changed to {\ttfamily Projective\+Point} (3d xyz)
\end{DoxyItemize}

ed25519:\+


\begin{DoxyItemize}
\item {\ttfamily Signature} was removed in favor of raw bytes
\item {\ttfamily get\+Shared\+Secret} was moved to {\ttfamily x25519} module
\end{DoxyItemize}

\begin{DoxyItemize}
\item Renamed Point\+G1 -\/\texorpdfstring{$>$}{>} G1.\+Point, Point\+G2 -\/\texorpdfstring{$>$}{>} G2.\+Point
\item Renamed Point\+G2.\+from\+Signature -\/\texorpdfstring{$>$}{>} Signature.\+decode, Point\+G2.\+to\+Signature -\/\texorpdfstring{$>$}{>} Signature.\+encode
