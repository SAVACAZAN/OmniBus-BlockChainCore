Lattice-\/based key encapsulation mechanism, defined in \href{https://nvlpubs.nist.gov/nistpubs/FIPS/NIST.FIPS.203.pdf}{\texttt{FIPS-\/203}} (\href{https://www.pq-crystals.org/kyber/resources.shtml}{\texttt{website}}, \href{https://github.com/pq-crystals/kyber}{\texttt{repo}}). Can be used as follows:\+


\begin{DoxyEnumerate}
\item {\itshape Alice} generates secret \& public keys, then sends public\+Key to {\itshape Bob}
\item {\itshape Bob} generates shared secret for Alice public\+Key. bob\+Shared never leaves {\itshape Bob} system and is unknown to other parties
\item {\itshape Alice} gets and decrypts cipher\+Text from Bob Now, both Alice and Bob have same shared\+Secret key without exchanging in plain\+Text:\+ alice\+Shared == bob\+Shared.
\end{DoxyEnumerate}

There are some concerns with regards to security:\+ see \href{https://blog.cr.yp.to/20231003-countcorrectly.html}{\texttt{djb blog}} and \href{https://groups.google.com/a/list.nist.gov/g/pqc-forum/c/W2VOzy0wz_E}{\texttt{mailing list}}. Old, incompatible version (Kyber) is not provided. Open an issue if you need it.

\begin{DoxyWarning}{Warning}
Unlike ECDH, KEM doesn\textquotesingle{}t verify whether it was "{}\+Bob"{} who\textquotesingle{}ve sent the ciphertext. Instead of throwing an error when the ciphertext is encrypted by a different pubkey, {\ttfamily decapsulate} will simply return a different shared secret. ML-\/\+KEM is also probabilistic and relies on quality of CSPRNG.
\end{DoxyWarning}

\doxylink{namespace_we}{We} suggest to use ECC + ML-\/\+KEM for key agreement, ECC + SLH-\/\+DSA for signatures.

ML-\/\+KEM and ML-\/\+DSA are lattice-\/based. SLH-\/\+DSA is hash-\/based, which means it is built on top of older, more conservative primitives. NIST guidance for security levels:\+
