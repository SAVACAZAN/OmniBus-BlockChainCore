\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md57}{}\doxysection{\texorpdfstring{CONTEXT}{CONTEXT}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md57}
I\textquotesingle{}m building {\bfseries{Omni\+Bus}}, a Layer-\/1 blockchain written in {\bfseries{Zig 0.\+15.\+2}} on Linux (testnet seed runs on a shared-\/\+CPU VPS, \texorpdfstring{$\sim$}{\string~}2 GHz invariant TSC, 2 cores, 1 GB RAM). The repo has \texorpdfstring{$\sim$}{\string~}80k LoC of Zig across {\ttfamily core/\+\texorpdfstring{$\ast$}{*}.zig}.\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md58}{}\doxysubsection{\texorpdfstring{Architecture (current, working)}{Architecture (current, working)}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md58}

\begin{DoxyItemize}
\item {\bfseries{Block model}}:\+ Bitcoin-\/style PoW with difficulty=1 on testnet. Each block targets {\bfseries{1 second}} wall-\/clock. Inside each block we run a 10-\/tick {\bfseries{sub-\/block}} loop (was 75ms × 10 = 750ms, now 0ms — sub-\/blocks are pure TX-\/batching, no real-\/time pacing).
\item {\bfseries{Slot leader rotation}}:\+ deterministic via {\ttfamily leader\+For\+Slot(slot\+\_\+id, prev\+\_\+block\+\_\+hash, validators)} — SHA-\/256 of (slot\+\_\+id \texorpdfstring{$\vert$}{|}\texorpdfstring{$\vert$}{|} prev\+\_\+hash) mod weighted-\/validator-\/set.
\item {\bfseries{Liveness fallback}}:\+ if leader is silent past {\ttfamily SLOT\+\_\+\+TIMEOUT\+\_\+\+MS} (300 ms by default, 50 ms when peer offline), the lex-\/smallest validator takes the slot.
\item {\bfseries{Single source of time}}:\+ {\ttfamily core/\+orchestrator.\+zig} exposes {\ttfamily Atomic\+Clock} (real or fake backend), {\ttfamily now\+Cycles()} (rdtscp inline asm on x86\+\_\+64), {\ttfamily binary\+Spectrum} (bit-\/level visualisation), {\ttfamily clock\+Score60s} (0-\/100 health metric for a 60s span), {\ttfamily calibrate\+Tsc\+Per\+Sec} (one-\/shot at startup → \texorpdfstring{$\sim$}{\string~}2 GHz on the VPS), {\ttfamily Time\+Orchestrator} (5 named timers:\+ slot, sub\+\_\+block, shard, ws\+\_\+exchange, stabilizer; with skip-\/when-\/late semantics so a 500ms GC pause produces 1 firing, not 12 catch-\/up firings).
\item {\bfseries{Slot\+Calendar}}:\+ ring of next 60 slots, deterministic leader per slot, refreshed states (future /\+ in\+\_\+flight /\+ finalized /\+ missed). Rebuilt every 10 blocks for amortised cost.
\item {\bfseries{Stabilizer}}:\+ per-\/block ring buffer of the last 3600 arrival timestamps. Reports 1-\/min and 60-\/min rolling rates every 60s and tunes a {\ttfamily timeout\+\_\+mult} ∈ \+[0.\+2, 2.\+0]\+ proportional to (observed/\+target).
\item {\bfseries{TCP\+\_\+\+NODELAY}} is now set on all P2P sockets (inbound + outbound).
\item {\bfseries{Burst smoothing}}:\+ minimum 100 ms between consecutive blocks produced by us, {\bfseries{only when we\textquotesingle{}re at-\/or-\/above target rate}}. Below target → free burst, no smoothing (lets the chain catch up after VPS scheduler pauses).
\item {\bfseries{DB persistence}}:\+ removed from the hot path entirely. The blockchain itself is the database; balances/\+nonces/\+pubkey-\/registry are deterministically replayed from chain on startup. Restart resyncs from peers.
\end{DoxyItemize}\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md59}{}\doxysubsection{\texorpdfstring{Cryptographic stack}{Cryptographic stack}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md59}

\begin{DoxyItemize}
\item secp256k1 ECDSA (pure Zig, \texorpdfstring{$\sim$}{\string~}80 KB), bech32 ({\ttfamily ob1q…} addresses)
\item BIP-\/32 HD wallets, BIP-\/39 mnemonic, optional 25th-\/word passphrase
\item liboqs PQ wrappers (ML-\/\+DSA-\/87, Falcon-\/512, SLH-\/\+DSA-\/256s, ML-\/\+KEM-\/768) for future migration
\item SHA-\/256 for block hashes + merkle roots; software impl, no Intel SHA-\/\+NI yet

\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md62}{}\doxysubsection{\texorpdfstring{Best ever observed (and lost)}{Best ever observed (and lost)}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md62}
The user reports having seen up to {\bfseries{58 blocks/\+min}} at one point, but they don\textquotesingle{}t remember exactly which commit. Looking at the history, the candidate window is between {\ttfamily 0606de8} (RDTSC + ws-\/feed-\/to-\/clock, 45-\/48/\+min observed) and {\ttfamily e2e5a2a} (calendar regression to 36/\+min).

The slot calendar in {\ttfamily e2e5a2a} cost \texorpdfstring{$\sim$}{\string~}12 blocks/\+min because it was calling {\ttfamily leader\+For\+Slot} (a SHA-\/256 hash) 60 times per block. The follow-\/up {\ttfamily 3104f5b} recovered most of that loss by throttling rebuild to every 10 blocks. \doxylink{namespace_we}{We} may still be a few blocks/\+min below the peak because the calendar rebuild is now amortised but still on the hot path.\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md63}{}\doxysubsection{\texorpdfstring{Best vs worst observed in a single run}{Best vs worst observed in a single run}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md63}

\item {\bfseries{Median latency}}:\+ 231 ms in the best windows

\end{DoxyItemize}\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md64}{}\doxysubsection{\texorpdfstring{Persistent problem}{Persistent problem}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md64}
Even with all the above, on the shared VPS the rate is {\bfseries{42–50/\+min, not 60}}. Inspection of the logs shows the cause is exclusively {\bfseries{hypervisor scheduler starvation}} — every few minutes the process is suspended for 5–18 seconds, during which no block can be produced. {\ttfamily clock\+\_\+score} drops to 47–60/\+100 in those windows; bursts fire as soon as we resume but the smoothing ceiling caps them.

This is {\bfseries{not a code problem}}. On dedicated hardware (e.\+g. a baremetal Linux machine or our planned Omni\+Bus OS) we expect to hit 60/\+min comfortably and the architecture supports much higher (the sub-\/block loop is 0 ms per tick now; each block is bounded by PoW at diff=1 plus state apply, \texorpdfstring{$\sim$}{\string~}50–200 µs typical).

But I want to validate the architecture itself. The current model is roughly:\+


\begin{DoxyEnumerate}
\item {\bfseries{PoW + slot-\/leader rotation}} (deterministic per slot, with liveness fallback to lex-\/min validator on timeout).
\item {\bfseries{Sub-\/blocks as pure TX batching}} (10 batches per slot, aggregate merkle root, no individual broadcast).
\item {\bfseries{Best-\/effort gossip with TCP\+\_\+\+NODELAY}} for low-\/latency block announces.
\item {\bfseries{Self-\/tuning timeout multiplier}} based on observed-\/vs-\/target rate.
\item {\bfseries{JSON-\/\+RPC 2.\+0}} + a Kraken-\/compatible REST surface for exchange semantics.
\end{DoxyEnumerate}\hypertarget{md__k_i_m_i___p_r_o_m_p_t_autotoc_md65}{}\doxysubsection{\texorpdfstring{What I want from you}{What I want from you}}\label{md__k_i_m_i___p_r_o_m_p_t_autotoc_md65}

I AM looking for designs where the {\bfseries{consensus + slot model itself}} is different, and where the trade-\/off makes sense for our use case (small validator set 2–10 nodes, 1 s target, JSON-\/\+RPC compatibility, PQ-\/ready signatures, eventual baremetal port).
