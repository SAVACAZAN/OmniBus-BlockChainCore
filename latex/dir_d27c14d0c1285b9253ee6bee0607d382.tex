\doxylink{namespace_we}{We} suggest to use {\bfseries{XCha\+Cha20-\/\+Poly1305}} because it\textquotesingle{}s very fast and allows random keys. {\bfseries{AES-\/\+GCM-\/\+SIV}} is also a good idea, because it provides resistance against nonce reuse. {\bfseries{AES-\/\+GCM}} is a good option when those two are not available.\hypertarget{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4551}{}\doxysubsubsection{\texorpdfstring{How to encrypt properly}{How to encrypt properly}}\label{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4551}

\begin{DoxyItemize}
\item Use unpredictable key with enough entropy
\begin{DoxyItemize}
\item Random key must be using cryptographically secure random number generator (CSPRNG), not {\ttfamily Math.\+random} etc.
\item Non-\/random key generated from KDF is fine
\item Re-\/using key is fine, but be aware of rules for cryptographic key wear-\/out and encryption limitsencryption limits
\end{DoxyItemize}
\end{DoxyItemize}
\begin{DoxyItemize}
\item Use new nonce every time and don\textquotesingle{}t repeat itdon\textquotesingle{}t repeat it
\end{DoxyItemize}
\begin{DoxyItemize}
\item chacha and salsa20 are fine for sequential counters that {\itshape never} repeat:\+ {\ttfamily 01, 02...}
\item xchacha and xsalsa20 can use random nonces instead
\item AES-\/\+GCM should use 12-\/byte nonces:\+ smaller nonces are security risk
\end{DoxyItemize}

Prefer authenticated encryption (AEAD)
\begin{DoxyItemize}
\item Good:\+ chacha20poly1305, GCM, GCM-\/\+SIV, Cha\+Cha+\+HMAC, CTR+\+HMAC, CBC+\+HMAC
\item Bad:\+ chacha20, raw CTR, raw CBC
\item Flipping bits or ciphertext substitution won\textquotesingle{}t be detected in unauthenticated ciphers
\item Polynomial MACs are not perfect for every situation:\+ they lack Random Key Robustness:\+ the MAC can be forged, and can\textquotesingle{}t be used in PAKE schemes. See \href{https://keymaterial.net/2020/09/07/invisible-salamanders-in-aes-gcm-siv/}{\texttt{invisible salamanders attack}}. To combat salamanders, {\ttfamily hash(key)} can be included in ciphertext, however, this would violate ciphertext indistinguishability:\+ an attacker would know which key was used -\/ so {\ttfamily HKDF(key, i)} could be used instead.
\end{DoxyItemize}

\begin{DoxyItemize}
\item For example, using ECDH key in AES can be bad
\item Use hkdf or, at least, a hash function to create sub-\/key instead
\end{DoxyItemize}\hypertarget{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4552}{}\doxysubsubsection{\texorpdfstring{Nonces}{Nonces}}\label{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4552}
Most ciphers need a key and a nonce (aka initialization vector /\+ IV) to encrypt a data. Repeating (key, nonce) pair with different plaintexts would allow an attacker to decrypt it. \begin{DoxyVerb}ciphertext_a = encrypt(plaintext_a, key, nonce)
ciphertext_b = encrypt(plaintext_b, key, nonce)
stream_diff = xor(ciphertext_a, ciphertext_b)    # Break encryption
\end{DoxyVerb}


One way of not repeating nonces is using counters:\+ \begin{DoxyVerb}for i in 0..:
    ciphertext[i] = encrypt(plaintexts[i], key, i)
\end{DoxyVerb}


Another is generating random nonce every time:\+ \begin{DoxyVerb}for i in 0..:
    rand_nonces[i] = random()
    ciphertext[i] = encrypt(plaintexts[i], key, rand_nonces[i])
\end{DoxyVerb}



\begin{DoxyItemize}
\item Counters are OK, but it\textquotesingle{}s not always possible to store current counter value:\+ e.\+g. in decentralized, unsyncable systems.
\item Randomness is OK, but there\textquotesingle{}s a catch:\+ Cha\+Cha20 and AES-\/\+GCM use 96-\/bit /\+ 12-\/byte nonces, which implies higher chance of collision. In the example above, {\ttfamily random()} can collide and produce repeating nonce. Chance is even higher for 64-\/bit nonces, which GCM allows -\/ don\textquotesingle{}t use them.
\item To safely use random nonces, utilize XSalsa20 or XCha\+Cha:\+ they increased nonce length to 192-\/bit, minimizing a chance of collision. AES-\/\+SIV is also fine. In situations where you can\textquotesingle{}t use e\+Xtended-\/nonce algorithms, key rotation is advised. hkdf would work great for this case.
\end{DoxyItemize}\hypertarget{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4553}{}\doxysubsubsection{\texorpdfstring{Encryption limits}{Encryption limits}}\label{dir_d27c14d0c1285b9253ee6bee0607d382_autotoc_md4553}

For non-\/deterministic (not ECB) schemes, initialization vector (IV) is mixed to block/\+key; and each new round either depends on previous block\textquotesingle{}s key, or on some counter.


\begin{DoxyItemize}
\item {\bfseries{ECB}} (Electronic Codebook):\+ Deterministic encryption; identical plaintext blocks yield identical ciphertexts. Not secure due to pattern leakage. due to pattern leakage. See \href{https://words.filippo.io/the-ecb-penguin/}{\texttt{AES Penguin}}
\item {\bfseries{CBC}} (Cipher Block Chaining):\+ Each plaintext block is XORed with the previous block of ciphertext before encryption. Hard to use:\+ requires proper padding and an IV. Unauthenticated:\+ needs MAC.
\item {\bfseries{CTR}} (Counter Mode):\+ Turns a block cipher into a stream cipher using a counter and IV (nonce). Efficient and parallelizable. Requires a unique nonce per encryption. Unauthenticated:\+ needs MAC.
\item {\bfseries{GCM}} (Galois/\+\+Counter Mode):\+ Combines CTR mode with polynomial MAC. Efficient and widely used. \item {\bfseries{SIV}} (Synthetic IV):\+ GCM with nonce-\/misuse resistance; repeating nonces reveal only the fact plaintexts are identical. Also suffers from GCM issues:\+ key wear-\/out limits \& MAC forging.
\item {\bfseries{XTS}}:\+ Designed for disk encryption. Similar to ECB (deterministic), but has {\ttfamily \+[i]\+\+[j]\+} tweak arguments corresponding to sector i and 16-\/byte block (part of sector) j. Lacks MAC.
