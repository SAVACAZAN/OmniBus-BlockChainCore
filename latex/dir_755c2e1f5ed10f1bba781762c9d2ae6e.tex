Perform blazing fastblazing fast equality comparisons (either deep or shallow) on two objects passed, while also maintaining a high degree of flexibility for various implementation use-\/cases. It has no dependencies, and is \texorpdfstring{$\sim$}{\string~}2kB when minified and gzipped.

The following types are handled out-\/of-\/the-\/box:\+


\begin{DoxyItemize}
\item Plain objects (including {\ttfamily react} elements and {\ttfamily Arguments})
\item Arrays
\item {\ttfamily Array\+Buffer} /\+ {\ttfamily Typed\+Array} /\+ {\ttfamily Data\+View} instances
\item {\ttfamily Date} objects
\item {\ttfamily Reg\+Exp} objects
\item {\ttfamily Map} /\+ {\ttfamily Set} iterables
\item {\ttfamily Promise} objects
\item Primitive wrappers ({\ttfamily new Boolean()} /\+ {\ttfamily new Number()} /\+ {\ttfamily new String()})
\item Custom class instances, including subclasses of native classes
\end{DoxyItemize}

Methods are available for deep, shallow, or referential equality comparison. In addition, you can opt into support for circular objects, or performing a "{}strict"{} comparison with unconventional property definition, or both. Caveats that impact the benchmark (and accuracy of comparison):\+


\begin{DoxyItemize}
\item {\ttfamily Map}s, {\ttfamily Promise}s, and {\ttfamily Set}s were excluded from the benchmark entirely because no library other than {\ttfamily deep-\/eql} fully supported their comparison
\item {\ttfamily fast-\/deep-\/equal}, {\ttfamily react-\/fast-\/compare} and {\ttfamily nano-\/equal} throw on objects with {\ttfamily null} as prototype ({\ttfamily Object.\+create(null)})
\item {\ttfamily assert.\+deep\+Strict\+Equal} does not support {\ttfamily NaN} or {\ttfamily Same\+Value\+Zero} equality for dates
\item {\ttfamily deep-\/eql} does not support {\ttfamily Same\+Value\+Zero} equality for zero equality (positive and negative zero are not equal)
\item {\ttfamily deep-\/equal} does not support {\ttfamily NaN} and does not strictly compare object type, or date /\+ regexp values, nor uses {\ttfamily Same\+Value\+Zero} equality for dates
\item {\ttfamily fast-\/deep-\/equal} does not support {\ttfamily NaN} or {\ttfamily Same\+Value\+Zero} equality for dates
\item {\ttfamily nano-\/equal} does not strictly compare object property structure, array length, or object type, nor {\ttfamily Same\+Value\+Zero} equality for dates
\item {\ttfamily react-\/fast-\/compare} does not support {\ttfamily NaN} or {\ttfamily Same\+Value\+Zero} equality for dates, and does not compare {\ttfamily function} equality
\item {\ttfamily shallow-\/equal-\/fuzzy} does not strictly compare object type or regexp values, nor {\ttfamily Same\+Value\+Zero} equality for dates
\item {\ttfamily underscore.\+is\+Equal} does not support {\ttfamily Same\+Value\+Zero} equality for primitives or dates
\end{DoxyItemize}

All of these have the potential of inflating the respective library\textquotesingle{}s numbers in comparison to {\ttfamily fast-\/equals}, but it was the closest apples-\/to-\/apples comparison I could create of a reasonable sample size. It should be noted that {\ttfamily react} elements can be circular objects, however simple elements are not; I kept the {\ttfamily react} comparison very basic to allow it to be included.\hypertarget{dir_755c2e1f5ed10f1bba781762c9d2ae6e_autotoc_md6602}{}\doxysubsection{\texorpdfstring{Development}{Development}}\label{dir_755c2e1f5ed10f1bba781762c9d2ae6e_autotoc_md6602}
