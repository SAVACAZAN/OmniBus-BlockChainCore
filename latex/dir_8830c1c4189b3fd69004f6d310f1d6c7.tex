\hypertarget{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4786}{}\doxysubsection{\texorpdfstring{Bech32, Bech32m and Bitcoin}{Bech32, Bech32m and Bitcoin}}\label{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4786}
\doxylink{namespace_we}{We} provide low-\/level bech32 operations. If you need high-\/level methods for BTC (addresses, and others), use \href{https://github.com/paulmillr/scure-btc-signer}{\texttt{scure-\/btc-\/signer}} instead.

Bitcoin addresses use both 5-\/bit words and bytes representations. They can\textquotesingle{}t be parsed using {\ttfamily bech32.\+decode\+To\+Bytes}.

Same applies to Lightning Invoice Protocol \href{https://github.com/lightning/bolts/blob/master/11-payment-encoding.md}{\texttt{BOLT-\/11}}. \doxylink{namespace_we}{We} have many tests in {\ttfamily ./\+test/\+bip173.test.\+js} that serve as minimal examples of Bitcoin address and Lightning Invoice Protocol parsers. Keep in mind that you\textquotesingle{}ll need to verify the examples before using them in your code.

Do something like this:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{const\ decoded\ =\ bech32.decode(address);}
\DoxyCodeLine{//\ NOTE:\ words\ in\ bitcoin\ addresses\ contain\ version\ as\ first\ element,}
\DoxyCodeLine{//\ with\ actual\ witness\ program\ words\ in\ rest}
\DoxyCodeLine{//\ BIP-\/141:\ The\ value\ of\ the\ first\ push\ is\ called\ the\ "{}version\ byte"{}.}
\DoxyCodeLine{//\ The\ following\ byte\ vector\ pushed\ is\ called\ the\ "{}witness\ program"{}.}

\hypertarget{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4787}{}\doxysubsection{\texorpdfstring{Design rationale}{Design rationale}}\label{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4787}
The code may feel unnecessarily complicated; but actually it\textquotesingle{}s much easier to reason about. Any encoding library consists of two functions:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{encode(A)\ -\/>\ B}
\DoxyCodeLine{decode(B)\ -\/>\ A}
\DoxyCodeLine{\ \ where\ X\ =\ decode(encode(X))}
\DoxyCodeLine{\ \ \#\ encode(decode(X))\ can\ be\ !==\ X!}
\DoxyCodeLine{\ \ \#\ because\ decoding\ can\ normalize\ input}
\DoxyCodeLine{}
\DoxyCodeLine{e.g.}
\DoxyCodeLine{base58checksum\ =\ \{}
\DoxyCodeLine{\ \ encode():\ \{}
\DoxyCodeLine{\ \ \ \ //\ checksum}
\DoxyCodeLine{\ \ \ \ //\ radix\ conversion}
\DoxyCodeLine{\ \ \ \ //\ alphabet}
\DoxyCodeLine{\ \ \},}
\DoxyCodeLine{\ \ decode():\ \{}
\DoxyCodeLine{\ \ \ \ //\ alphabet}
\DoxyCodeLine{\ \ \ \ //\ radix\ conversion}
\DoxyCodeLine{\ \ \ \ //\ checksum}
\DoxyCodeLine{\ \ \}}
\DoxyCodeLine{\}}

\end{DoxyCode}


But instead of creating two big functions for each specific case, we create them from tiny composable building blocks:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{base58checksum\ =\ chain(checksum(),\ radix(),\ alphabet())}

\end{DoxyCode}


Which is the same as chain/\+pipe/\+sequence function in Functional Programming, but significantly more useful since it enforces same order of execution of encode/\+decode. Basically you only define encode (in declarative way) and get correct decode for free. So, instead of reasoning about two big functions you need only reason about primitives and encode chain. The design revealed obvious bug in older version of the lib, where xmr version of base58 had errors in decode\textquotesingle{}s block processing.

Besides base-\/encodings, we can reuse the same approach with any encode/\+decode function ({\ttfamily bytes2number}, {\ttfamily bytes2u32}, etc). For example, you can easily encode entropy to mnemonic (BIP-\/39):\+

\hypertarget{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4788}{}\doxysubsubsection{\texorpdfstring{base58 is O(n\texorpdfstring{$^\wedge$}{\string^}2) and radixes}{base58 is O(n\texorpdfstring{$^\wedge$}{\string^}2) and radixes}}\label{dir_8830c1c4189b3fd69004f6d310f1d6c7_autotoc_md4788}
{\ttfamily Uint8\+Array} is represented as big-\/endian number:\+


\begin{DoxyCode}{0}
\DoxyCodeLine{[1,\ 2,\ 3,\ 4,\ 5]\ -\/>\ 1*(256**4)\ +\ 2*(256**3)\ 3*(256**2)\ +\ 4*(256**1)\ +\ 5*(256**0)}
\DoxyCodeLine{where\ 256\ =\ 2**8\ (8\ bits\ per\ byte)}

\end{DoxyCode}


which is then converted to a number in another radix/\+base (16/\+32/\+58/\+64, etc).

However, generic conversion between bases has \href{https://cs.stackexchange.com/q/21799}{\texttt{quadratic O(n\texorpdfstring{$^\wedge$}{\string^}2) time complexity}}.

Which means base58 has quadratic time complexity too. Use base58 only when you have small constant sized input, because variable length sized input from user can cause DoS.

On the other hand, if both bases are power of same number (like {\ttfamily 2\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}8 \texorpdfstring{$<$}{<}-\/\texorpdfstring{$>$}{>} 2\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}64}), there is linear algorithm.
