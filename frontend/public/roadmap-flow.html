📐 Pyramid · How to read

  • Filled tier — chain has reached that order of magnitude
  • Partial tier — currently filling, opacity ∝ progress
  • Dim tier — block range not yet mined
  • Top apex (1%) — the infinity gap. The chain has no hard cap (halving forever, dust block rewards), so the crown of the pyramid stays open as a permanent reminder that the chain keeps growing.
  • Logarithmic scale — the 7 main tiers are not equal block counts; they map log₁₀(height) so the visual stays alive whether the chain is at 100 or at 100 million blocks
  • 📊 Block height — total blocks finalised on chain
  • 🎯 % filled — log-scaled progress against the 99% region
  • ⏱ Last block age — seconds since latest block; healthy < 3s

⏳ Hourglass · How to read

  • Top sand — time remaining in the current 60s cycle
  • Bottom sand — time elapsed; resets each cycle
  • Slot phase — Ns / 60s, the wall-clock progress through the cycle
  • Mempool size — pending TXs awaiting the next block
  • RDTSC cycles — direct CPU cycle counter (rdtscp instruction). On a 3GHz core: ~3 billion ticks/sec
  • Cycles per second — measured rdtsc delta over the last 1s; baseline for clock health
  • Slot calendar — count of pre-computed future slots awaiting fill
  • Spectrum line — 64 bits of the cycle counter, MSB→LSB. Stable high bits = healthy clock; broken patterns = scheduler jitter

🔧 Why two hourglass models?

  • Glass model — minimalist, scales cleanly on small screens
  • Wood model — classic feel, full SVG with wood grain + glass highlights
  • Pick whichever you prefer; both poll the same RPC and share the same cycleStart reference
  • Page is pure static HTML + inline SVG + vanilla JS — no React, no WS subscription, no allocator pressure on the node
Read-only view; polls RPC every 1s. Does not subscribe to WebSocket or hold long-lived state — designed to add near-zero load on the node. Pyramid tier = ⌊height / 10000⌋. Hourglass cycle = 60s.
